# 角色卡管理器 — 设计与集成文档

## 一、定位

角色卡管理器不再按“独立 HTML + localStorage 小工具”实现，而是作为当前项目中的正式功能模块集成到现有模组系统。

必须遵循当前项目主线约束：

- 数据正式落在后端，不走前端静默本地保存
- 接口由后端提供，前端通过统一服务层调用
- 新接口应纳入 Huma/OpenAPI/heyapi/TanStack Query 主链
- 导入导出要纳入现有备份包体系
- 权限沿用当前模组成员体系：`GM / 副GM / PL`

角色卡管理器的目标不是替代现有“角色 / 怪物”实体，而是提供一套更细粒度、可直接跑团使用的系统化角色卡文档。

## 二、与现有系统的关系

### 1. 与现有角色实体的边界

- 现有 `character / monster` 仍然保留，继续承担剧情设定、关系图、时间线引用等世界观信息功能
- 新增“角色卡”作为独立文档类型，专注于规则数据、数值、装备、法术、状态、导入导出
- 角色卡可选地“关联到某个现有实体”，但不强制绑定
- 一个实体可以关联多张角色卡，例如：
  - NPC 不同时期的不同卡面
  - 玩家角色升级前后的不同版本
  - 怪物模板与实战实例

### 2. 与协作体系的关系

- 角色卡属于模组内正式数据
- GM 与副GM默认可查看、编辑、删除全部角色卡
- PL 是否可查看、编辑某张角色卡，建议按“卡级权限”控制
- 角色卡编辑应复用现有协作设计：
  - 显式保存或自动保存
  - 版本号
  - 冲突提示
  - 可选租约

### 3. 与备份导入导出的关系

- 单模组导出时，角色卡要随模组一并导出
- 全量导出时，角色卡要纳入所有模组的备份包
- 若角色卡引用了图片资源，应复用现有 `assets/graph_assets/...` 导出链路

## 三、一期范围

一期建议只做以下正式范围：

- 支持 `CoC` 与 `DND5e`
- 支持多角色卡列表
- 支持卡片创建、复制、删除、重命名
- 支持按系统展示不同字段
- 支持文本导入 / 文本导出
- 支持头像与图片资源引用
- 支持多卡对比视图
- 支持与现有实体建立“弱关联”

一期暂不强求：

- 完整规则自动校验
- 所有职业/子职业/法术库的规则内置
- 完整拖拽式卡面排版编辑器
- 离线独立运行
- 完全自动的复杂公式依赖链

## 四、页面结构建议

### 1. 入口

- 在模组内导航新增 `角色卡管理器`
- 页面路径建议为 `/character-sheets`
- 后续若要支持直接从某个角色实体跳转，可增加“打开关联角色卡”按钮

### 2. 页面布局

- 左侧：角色卡列表、系统筛选、搜索、创建按钮
- 中间：当前角色卡主体编辑区
- 右侧：预览/计算结果/导入导出/多卡对比入口

### 3. 详情模式

- 默认预览
- 点击“开始编辑”进入编辑模式
- 编辑结束后保存
- 与当前项目其他详情页交互保持一致，避免再次出现“点开即编辑”的体验问题

## 五、权限设计

### 1. 模组级权限

- `GM`：全部权限
- `副GM`：除“删除模组、管理其他管理者”外，与 GM 一致
- `PL`：仅能访问被授权的角色卡

### 2. 角色卡级权限

建议每张角色卡增加独立可见性：

- `owner_only`：仅 GM / 副GM
- `party_read`：所有模组成员可查看，PL 不可编辑
- `party_edit`：所有模组成员可编辑
- `assigned_only`：仅指定成员可查看 / 编辑

### 3. 卡片负责人

建议增加：

- `ownerUserId`
- `ownerUsername`
- `assignedUserIds`

用于支持“这是某个 PL 的玩家卡，但 GM 也可管理”的场景。

## 七、关键计算建议

### 1. DND5e

- 属性调整值：`Math.floor((score - 10) / 2)`
- 熟练加值：基于等级自动推导，但允许手工覆盖
- 技能总值：属性调整值 + 熟练加值 × 熟练系数
- 法术 DC：`8 + 熟练加值 + 施法属性调整值`
- 法术攻击加值：`熟练加值 + 施法属性调整值`
- 负重统计可做“辅助展示”，不要一上来做成强校验

### 2. CoC

- HP / SAN / MP 支持“当前 / 最大”拆分存储
- 伤害加值、体格、移动率可以自动算出默认值，但允许手动覆盖
- 技能值默认模板与自定义技能并存

### 3. 原则

- 自动计算只提供默认建议
- 用户手动改过的值应保留，不要被自动逻辑反复覆盖

## 八、API 设计建议

以下接口建议纳入 Huma/OpenAPI，并由 heyapi 生成客户端。

### 1. 列表与详情

- `GET /api/v2/campaigns/{campaignId}/character-sheets`
  - 返回角色卡摘要列表
- `POST /api/v2/campaigns/{campaignId}/character-sheets`
  - 创建角色卡
- `GET /api/v2/campaigns/{campaignId}/character-sheets/{sheetId}`
  - 获取详情
- `PUT /api/v2/campaigns/{campaignId}/character-sheets/{sheetId}`
  - 保存角色卡
- `DELETE /api/v2/campaigns/{campaignId}/character-sheets/{sheetId}`
  - 删除角色卡

### 2. 附加操作

- `POST /api/v2/campaigns/{campaignId}/character-sheets/{sheetId}/duplicate`
  - 复制一张角色卡
- `POST /api/v2/campaigns/{campaignId}/character-sheets/{sheetId}/link-entity`
  - 绑定/解绑现有实体
- `POST /api/v2/campaigns/{campaignId}/character-sheets/{sheetId}/import-text`
  - 文本导入并返回解析结果
- `GET /api/v2/campaigns/{campaignId}/character-sheets/{sheetId}/export`
  - 导出为文本或 JSON
- `GET /api/v2/campaigns/{campaignId}/character-sheets/compare?ids=a,b,c`
  - 拉取多卡对比数据

### 3. 导入接口

建议导入走两段式：

- 第一步：`POST /import-text/preview`
  - 只解析，不落库
  - 返回识别到的系统、字段映射、警告、缺失项
- 第二步：`POST /import-text/commit`
  - 用户确认后再写入正式数据

这样能避免“误识别但直接写库”的风险。

### 4. 响应建议

所有写接口建议统一返回：

```ts
interface CharacterSheetWriteResponse {
  sheet: CharacterSheetDocument;
  warnings?: string[];
}
```

导入预览建议返回：

```ts
interface CharacterSheetImportPreview {
  detectedSystem: 'coc7' | 'dnd5e' | 'unknown';
  confidence: number;
  warnings: string[];
  missingFields: string[];
  normalizedPayload: Partial<CoCSheetPayload | Dnd5eSheetPayload>;
}
```

## 十、导入导出建议

### 1. 文本导入

支持：

- CoC `.st`
- DND `.dst`
- 常见中文自然语言片段

但一期不要承诺“任意机器人格式都能 100% 识别”。

### 2. JSON 导出

建议提供：

- 角色卡单卡 JSON
- 模组内全部角色卡 JSON
- 备份包内自动包含角色卡 JSON

### 3. 图片处理

- 头像优先引用现有资源管理路径
- 避免把大图直接塞进 JSON
- 备份导出时按资源路径收集图片

## 十一、版本与冲突

角色卡属于协作数据，建议从一开始就带：

- `version`
- `updatedAt`
- `updatedBy`
- 冲突返回
- 可选租约

这样可以直接复用团队笔记、任务看板已经验证过的交互模式。

## 十二、当前文档缺口补充

原始设计里还缺少以下正式集成必需项：

- 没有后端数据模型
- 没有模组权限设计
- 没有 API 设计
- 没有和现有角色/怪物实体的边界
- 没有导入预览/确认机制
- 没有备份包集成方案
- 没有图片资源引用策略
- 没有版本冲突与协作编辑方案

这些已经在本版文档中补齐为首轮可实现方案。

## 十三、建议实现顺序

1. 先落后端数据模型与 Huma/OpenAPI 契约
2. 再生成前端客户端与 Query hooks
3. 先做列表页 + 详情页 + 创建/删除 + 保存
4. 再接入 CoC / DND 字段编辑器
5. 再做导入预览
6. 最后接多卡对比、文本导出、实体关联

## 十四、结论

角色卡管理器适合做成“模组内正式文档系统”，而不是单独的前端玩具页。

这样做的好处是：

- 权限清晰
- 数据可备份
- 可协作
- 可进入版本记录
- 可与现有角色、怪物、资源系统融合

后续开始实现时，应以本文件中的“后端文档模型 + API + Query 接入 + 分系统字段组件化”作为主线。
